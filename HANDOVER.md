It broke outright on a date where every upgraded candidate was risk-blocked; fuzz-testing across 28 simulated dates (2025–2027) then surfaced the second, rarer failure mode — some dates have **zero** WATCH candidates in the labelled shortlist at all, leaving nothing to upgrade. Rewrote to directly construct a single WATCH candidate with fixed, schema-valid entry/stop/target and call `_resolve_decision`/`_handle_proposal` directly (same pattern as `conftest.py`'s `inject_pending_proposal`), eliminating both dependencies — verified deterministic across all 28 simulated dates. **This is the second time the "mock market data is date-seeded" risk (§7.6) has silently broken a merged test** (first was `test_mfe_mae.py`, PR #22) — treat it as a recurring hazard class, not a one-off; see §7.6 for the generalized guidance this incident adds.
- **PR7 — TQS v0 Shadow Scoring** (`b602da8`). Adds a deterministic, explainable **Trade Quality Score**: a weighted composite over 7 components (reward:risk geometry, interest strength, microstructure quality, AI conviction, labeller conviction, narrative/catalyst alignment, event-risk clearance; weights sum to 100, interest_strength 25 / ai_conviction 20 / reward_risk 15 / the rest 10 each) computed for every scanned candidate and separately for every proposal, multiplied by a `data_confidence` factor so missing evidence can only lower the score, never raise it. It is an **attention-worthiness ranking signal**, explicitly not a probability/expected-return/sizing signal — built to be validated later against `candidate_outcomes`/`trade_outcomes` (did high-TQS trades actually perform better?), not yet scheduled as its own PR. **Shadow-only by construction, not discipline**: `orchestrator.py` calls `tqs.score_scan_batch()` only *after* logging "scan_once complete" — i.e. strictly after every decision that scan already committed — writing to a separate `tqs_scores` table with no read path from any risk/approval/execution code. Version-pinned (`TQS_VERSION="0.1.0"`); never retro-rescored. One flag, `TQS_SHADOW_ENABLED` (default **true**, zero-cost pure computation; disabling only skips the shadow write). +76 tests (`test_tqs_scoring.py` 45 pure-formula, `test_tqs_flow.py` 31 end-to-end). One test-hygiene fix pre-merge (`4f8bf76`): a weak early-exit test that skipped its own assertions if the mock scan didn't naturally produce a proposal, rewritten deterministically via direct construction (same class of issue as this checkpoint's fix, above, but caught and fixed before merge that time).
- **PR6 — Proposal TTL / Stale-Approval Guard** (`cdf0dcc`). Every proposal is now stamped at creation with a TTL + `expires_at_utc`, frozen from the market session active at that instant (RTH default 1800s/30min; premarket+afterhours share a shorter 300s/5min bucket; CLOSED gets 0s/immediate) — solves a proposal from an earlier scan sitting "pending approval" indefinitely and getting approved long after the market conditions that justified it have changed. TTL is computed once at creation, never recomputed at approval time. Same PR also ships **same-symbol supersession**: a fresh approvable proposal marks any OTHER still-open proposal for that symbol as `superseded` (never deleted, fully auditable) — deliberately only from the risk-*approved* branch, so a risk-blocked re-evaluation never strips optionality from an existing good proposal. **HIGH-severity pre-merge audit finding, adversarially proven** (`35fcd69`): the stale-approval guard lived only inside `approve_proposal()` (the manual path) — the **AUTO-approval path** (`_handle_proposal → approvals.consider() → _execute()`) never called `approve_proposal()` at all and so bypassed TTL entirely; the audit proved this by producing **5 fills from proposals born already expired**. Rated HIGH not BLOCKER only because unreachable under default config (manual approval is default; AUTO is off-by-default paper-only; mock sessions always fix to REGULAR/1800s TTL) — but the PR's invariant ("no stale proposal can be approved") is unconditional, so it was closed regardless: `_execute()` — the single chokepoint every execution route funnels through — now checks `proposal_ttl.is_expired(...)` before any broker submission and blocks with `PROPOSAL_EXPIRED` if so; `_handle_proposal`'s auto branch also pre-checks TTL for a clean `expired` status instead of a generic `blocked` one. Config: `PROPOSAL_TTL_RTH_SECONDS` (1800, [60,7200]), `PROPOSAL_TTL_EXTENDED_HOURS_SECONDS` (300, [0,3600]), `PROPOSAL_TTL_CLOSED_SESSION_SECONDS` (0, [0,3600]). +54 tests (`test_proposal_ttl.py` 17, `test_proposal_ttl_flow.py` 37, incl. regression tests for the HIGH finding).
